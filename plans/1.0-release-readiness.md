# merTools 1.0 Release Readiness Plan

*Created: 2026-04-18. Updated: 2026-04-20. Branch of origin: `v0.9.0` @ `475e09f`.*

A 1.0 release is an API-stability commitment. The items below were surfaced
during a codebase review on 2026-04-18 (items 1–21) and a GitHub-issue
triage on 2026-04-20 (items 22–29). They are organized by blocker severity
and suggested sequencing. The status of each item should be updated as work
progresses.

**Open GitHub issues covered by this plan:** #78, #83, #84, #85, #101,
#116, #118, #121, #124, #134. Issues #106 and #131 are addressed by
v0.9.0 (PR #135) and are not listed here.

---

## Must-fix before 1.0

These are the items that are hardest (or impossible) to change after a 1.0
tag because they affect the public API surface, the numerical contract, or
ship-blocking CRAN/revdep expectations.

### 1. Drop or wire up `predictInterval(.paropts)`

- **Where:** `R/merPredict.R:26`, `R/merPredict.R:52`
- **Problem:** argument is documented as "not used - placeholder for future
  foreach options". Shipping a placeholder in 1.0 locks it into the API.
- **Decision:** either implement it (passing opts through to `foreach`) or
  remove it from the signature and update docs/tests.
- **Status:** open

### 2. Clean up `R/zzz.R`

- **Where:** `R/zzz.R`
- **Problems:**
  - `sum.mm()` has "Title" placeholder roxygen (line 17), three inherited
    lme4 FIXMEs, and no callers in the package.
  - `plot_sim_error_chks()` validates inputs and returns nothing — dead
    code in its current form.
  - `zzz()` at line 11 is a no-op stub.
- **Decision:** delete or properly finish. A 1.0 shouldn't contain orphaned
  lme4-port helpers.
- **Status:** open

### 3. Fix broken `class()` comparison in `predictInterval()` — DONE

- **Where:** `R/merPredict.R:60`
- **Problem:**
  ```r
  if (any(c("data.frame") != class(newdata))) { ... }
  ```
  For tibbles, `class(newdata)` has length 3. The comparison produces a
  length-3 logical and only "works" because `any()` happens to trigger the
  right branch. This is brittle and mis-signals intent.
- **Fix:** replace with `if (!inherits(newdata, "data.frame"))`.
- **Status:** done — `R/merPredict.R:63`

### 4. Add lifecycle badges to exported functions

- **Where:** all exported functions (40 across 12 files)
- **Problem:** no `@lifecycle` / `lifecycle_badge()` annotations exist. Users
  of a 1.0 release have no way to distinguish stable API from experimental
  additions (e.g. `REmargins` is newer and less battle-tested than `REsim`).
- **Action:**
  - Add `lifecycle` to `Imports:`.
  - Annotate each exported function with `stable`, `experimental`, or
    `deprecated`.
  - Deprecate (rather than remove) anything you don't want to commit to
    long-term support.
- **Status:** open

### 5. Resolve `residDF.merMod` scale-parameter TODO

- **Where:** `R/helpers.R:20-26`
- **Problem:** TODO reads: "how do we feel about counting the scale
  parameter???" This function feeds `simulate_residual_variance()` in the
  `predictInterval()` numerical pipeline. The answer is part of the 1.0
  numerical contract — must be decided before pinning.
- **Decision required:** include scale parameter in the DF subtraction
  (current behavior) or don't. Document the choice and the rationale in
  the docstring.
- **Status:** open

### 6. Reverse-dependency checks

- **Tool:** `revdepcheck::revdep_check()`
- **Why:** major version bumps are expected by CRAN to run revdep checks
  against the current public version.
- **Status:** open

## Should-fix before 1.0

Quality/hygiene items that won't block a tag technically but will embarrass
us if left undone in a 1.0 release.

### 7. Refresh vignettes (closes #116)

- **Files:** `vignettes/Using_predictInterval.Rmd`,
  `vignettes/merToolsIntro.Rmd`, `vignettes/marginal_effects.Rmd`,
  `vignettes/imputation.Rmd`
- **Problem:** all four are dated `2020-06-22`. Readers will see the
  package as stale-since-2020 regardless of actual activity.
- **#116 items to fix during the refresh:**
  - Header formatting issues in the main vignette.
  - The graphic comparing `predictInterval()` to `bootMer()` no longer
    lines up with the accompanying narrative (intervals don't cover as
    the text claims). Re-run, verify coverage, update graphic and/or
    narrative so they agree.
- **Action:** update dates, verify code still runs against current lme4 /
  dplyr / broom.mixed, re-knit via the `.Rmd.orig` pattern (already in
  place via `vignettes/precompile.R`).
- **Status:** open

### 8. Import hygiene

- **Where:** `NAMESPACE` (generated from `@import` tags in `R/`)
- **Problem:** `import(arm)`, `import(dplyr)`, `import(ggplot2)`,
  `import(lme4)` bring in the entire namespace of each. Creates collision
  risk and makes upstream deprecations dangerous.
- **Action:** convert each to specific `@importFrom` tags listing only
  symbols actually used. `dplyr` alone would likely shrink to ~5-10
  symbols.
- **Status:** open

### 9. Dependency surface review

- `arm` is in `Depends:` unversioned. Package is thinly maintained; if it
  breaks, merTools 1.0 breaks. Consider moving to `Imports:` with specific
  `importFrom(arm, sim, sigma.hat)` and document the binding.
- `shiny` in `Imports:` for the single `shinyMer()` function is heavy.
  Move to `Suggests:` with a `requireNamespace("shiny", quietly = TRUE)`
  runtime guard at the top of `shinyMer()`.
- `parallel` in `Suggests:` — this is a base R package. Drop.
- **Status:** open

### 10. Update `README.Rmd`

- **Where:** `README.Rmd`
- **Problem:** "Recent Updates" section stops at `0.6.4 (January 2026)`.
  No 0.9.x or 1.0 entries.
- **Action:** add entries and re-knit to `README.md`.
- **Status:** open

### 11. NEWS framing for 1.0

- **Where:** `NEWS.md`
- **Problem:** version jump from 0.6.5 → 0.9.0 → 1.0 is non-obvious.
  Users benefit from a short paragraph explaining what 1.0 means
  (API stability pledge, minimum R/lme4 versions, breaking changes,
  deprecations).
- **Status:** open

### 12. roxygen markdown mode (decide, don't defer)

- **Where:** `DESCRIPTION`
- **Problem:** `RoxygenNote: 7.3.3` but no `Roxygen: list(markdown = TRUE)`
  directive. Existing `.Rd` files use mixed LaTeX-style (`\url{}`,
  `\code{}`, `\emph{}`). Entering 1.0 without a deliberate choice locks
  in the current ambiguity.
- **Options:**
  - Stay LaTeX: do nothing but document the choice.
  - Enable markdown: add directive, regenerate all `.Rd`, review diffs.
- **Status:** open

### 13. `REmargins()` TODOs

- **Where:** `R/REmargins.R:102`, `:142`, `:161`
- **Items:**
  - `# TODO - how does performance scale to a large number of observations?` —
    scale question; benchmark and either fix or document the recommended
    input-size bounds.
  - `# TODO - how to break ties here?` — correctness-adjacent. Document
    the tie-breaking behavior explicitly in the function docstring.
  - `# TODO use future here` — the `.parallel = TRUE` branch still uses
    `foreach`. Modern idiom is `future.apply` (already in Suggests).
- **Status:** open

### 14. CITATION file

- **Where:** `inst/CITATION` (does not exist)
- **Why:** users publishing with merTools need a canonical cite. This is
  standard for 1.0 releases of packages used in research.
- **Status:** open

### 15. Full release-checklist triad

- `urlchecker::url_check()` — find broken/redirected URLs in docs.
- `spelling::spell_check_package()` — find typos in docs/vignettes.
- `devtools::check(remote = TRUE, manual = TRUE)` — includes the manual
  build, which CRAN runs.
- None of these have been run on the v0.9.0 branch to date.
- **Status:** open

### 16. Measure test coverage

- **Tool:** `covr::package_coverage()`
- **Target:** 80%+.
- **Expected cold spots:** `R/zzz.R`, `R/parallel.R`, `shinyMer.R`, and
  the `Amelia` / `rstanarm` / `future.apply` integration paths.
- **Status:** open

## Nice-to-have / post-1.0

Items that would be polish improvements but should not block the tag.

### 17. Cross-platform Layer 2 snapshot

- Current state: Layer 2 snapshot is Linux-only (documented in
  `tests/testthat/README.md`), because macOS-ARM64 and Windows produce
  BLAS-level numeric drift.
- Post-1.0 enhancement: add a macOS-latest snapshot with widened
  tolerance so cross-platform drift is at least observable rather than
  invisible.

### 18. pkgdown site verification

- Confirm the pkgdown GitHub Actions workflow builds cleanly against the
  0.9.x / 1.0 changes before tagging.

### 19. Modernize input validation to cli/rlang

- Current code uses bare `stop()` / `warning()` / `message()`. For 1.0
  polish (not a blocker), adopt `cli::cli_abort()` / `cli::cli_warn()`
  with custom error classes. Enables programmatic error handling by
  downstream code.

### 20. Replace `foreach` with `future.apply`

- If `.parallel` / `.paropts` are kept (see item 1), swap the backend to
  `future.apply` to match modern R parallelism idioms. Resolves the
  `REmargins.R` "use future here" TODO as a side-effect.

### 21. Repository housekeeping

- Add `*.tar.gz` (or at minimum `merTools_*.tar.gz`) to `.gitignore`.
- Remove the stray `merTools_0.9.0.tar.gz` artifact currently untracked
  in the working tree.
- Consider adding `^plans$` to `.Rbuildignore` so this doc doesn't ship
  to CRAN.
- **Status:** done — `*.tar.gz` added to `.gitignore`; `.Rbuildignore` already
  had `^plans$` and `^merTools.*\.tar\.gz$`; `merTools_0.9.0.tar.gz` is now
  gitignored (delete from disk when convenient).

## Open GitHub issues to resolve before 1.0

### 22. `predictInterval()` crash bug on no-intercept random-slope models (#101)

- **Issue:** `Error in coefs[, 1:colIdx, lvl] : subscript out of bounds`
  when predicting from models like
  `lmer(y ~ Days:Test + (0 + Days | Subject), ...)`.
- **Status notes:** NEWS for 0.5.0 claims this was fixed, but the issue
  is still open on GitHub. Must-fix because it's a crash in the flagship
  function on a legitimate (if unusual) model spec.
- **Action:** reproduce against `v0.9.0` with the reporter's sleepstudy
  example. If already fixed, close the issue with a confirming comment
  and add the example as a regression test. If still broken, fix in the
  `simulate_random_effects()` path and add the test.
- **Status:** open

### 23. `predictInterval()` crash on `reMatrix` with multiple RE terms (#118)

- **Where:** `R/predictInterval_helpers.R` around the `reMatrix`
  handling (formerly `merPredict.R:177`).
- **Problem:** `Error in dimnames(reMatrix) <- *vtmp* : 'dimnames' applied
  to non-array` when a grouping factor has 23+ levels and there's more
  than one random-effect term.
- **Why must-fix:** crash bug on a supported model shape. Locks in the
  predictInterval numerical pipeline before 1.0.
- **Action:** reproduce with reporter's `reMatrix` shape, add regression
  test under `test-predictInterval-invariants.R`, fix the dimension
  handling.
- **Status:** open

### 24. Non-reproducible `predictInterval()` with mixed observed/unobserved levels (#124)

- **Issue:** Multiple calls to `predictInterval()` return significantly
  different intervals (more than sampling noise) when `newdata` contains
  a mix of nested-factor levels that are and aren't in the training
  data. Per-row vs bulk calls also disagree.
- **Why must-fix:** affects the 1.0 numerical contract. The 0.9.0 RNG
  stream fix pinned reproducibility *with a seed* but didn't address
  this specific bulk-vs-row divergence.
- **Action:** reproduce from the issue transcript, determine whether the
  divergence comes from `mkNewReTrms()` handling of partial nesting, the
  `reMatrix` code path, or the fixed-effect simulation. Fix or, if the
  behavior is by design, document it clearly in `?predictInterval`.
- **Status:** open

### 25. `averageObs()` fails on two-column GLM specs (#83)

- **Issue:** `averageObs(gm1)` errors on
  `glmer(cbind(incidence, size - incidence) ~ period + (1|herd), ...)`
  because the model frame names don't line up with the original data
  when the LHS is a matrix.
- **Why must-fix:** `averageObs()` feeds `predictInterval()` and the
  `draw`/`wiggle` pipeline. Broken for any binomial two-column spec,
  which is a very common GLMM shape.
- **Action:** handle matrix-LHS formulas in `averageObs()` — extract the
  response terms from the formula rather than pattern-matching against
  model-frame columns.
- **Status:** open

### 26. `nlmer` support (#121)

- **Issue:** `predictInterval()` dies with
  `'family' is not a valid field or method name for reference class nlsResp`
  when passed an `nlmerMod`.
- **Decision required:** either (a) add minimal `nlmer` support, or
  (b) detect `nlmerMod` up front and raise a clean
  `stop("predictInterval() does not support nlmer models")`. Either
  way, the current cryptic S4 error is unacceptable for a 1.0 release.
- **Action:** short-term — add early class guard with clean error.
  Longer-term — evaluate what it would take to actually simulate from
  an `nlmer` model.
- **Status:** done — `R/merPredict.R:57-59` guards with clean `stop()`;
  longer-term nlmer support deferred to post-1.0.

### 27. `fastdisp()` should return its output (#134)

- **Issue:** `fastdisp()` prints to the console but doesn't return the
  summary object, so users can't store it.
- **Why pre-1.0:** changing a function's return value post-1.0 is a
  breaking change. Decide and document now.
- **Action:** return the summary invisibly (`invisible(x)` with `x` the
  summary object) while preserving the existing print behavior. Update
  docs and add a test asserting the returned object shape.
- **Status:** done — both methods already returned `invisible(out)`; updated
  `@return` docs and added `tests/testthat/test-fastdisp.R` (11 passing tests).

### 28. Add model-summary landing tab to `shinyMer()` (#78)

- **Issue:** user request for a "model info" landing page in the Shiny
  app: RE var/cov per level, fixed coefficients, total variance,
  REML/ML + optimizer info, ICC.
- **Why should-fix pre-1.0 (not post-1.0):** `shinyMer()` UI is part of
  the public API; adding a tab post-1.0 is fine, but redesigning the
  landing page later is noisier. Worth deciding the default view before
  locking the API.
- **Action:** add a "Model Summary" tab that surfaces
  `modelInfo(merMod)` plus `VarCorr()`, ICC (already computed elsewhere
  in package), optimizer settings from `merMod@optinfo`. Set it as the
  default tab.
- **Status:** open

### 29. REimpact / FE-RE plot polish (post-1.0: #84, #85)

- **#84:** add an option to plot multiple REimpact outputs on the same
  chart so grouping factors can be compared side-by-side.
- **#85:** improve the visual quality of REimpact and FE/RE plots —
  `mcmc_areas_data()` + `ggridges::geom_ridge_density()` were suggested
  by @tjmahr as a prototype path.
- **Why post-1.0:** purely cosmetic / additive. Shouldn't block the tag.
- **Status:** open — deferred to 1.1.

## Suggested sequencing

### Phase 1 — API lockdown (pre-RC)

Items 1, 2, 3, 5, 8, 22, 23, 24, 25, 26, 27. Hardest to change post-1.0:
public signature, namespace surface, numerical pipeline, crash bugs in
`predictInterval()` / `averageObs()`, and API decisions (`fastdisp` return
value, `nlmer` guard).

### Phase 2 — Commitments and reviews (RC candidate)

Items 4, 6, 9, 13, 16, 28. Lifecycle badges, revdep check, dependency
decisions, correctness TODOs, coverage measurement, `shinyMer` landing-tab
decision.

### Phase 3 — Release polish

Items 7 (+ #116), 10, 11, 12, 14, 15. Vignettes (now including the
`bootMer` graphic alignment), README, NEWS framing, roxygen markdown
decision, CITATION, final CRAN-style check.

### Phase 4 — Deferable / post-1.0

Items 17–21, 29. Includes cosmetic plot polish (#84, #85).

Items 1–6 and 22–27 are the set I'd want closed before cutting even an
RC.
